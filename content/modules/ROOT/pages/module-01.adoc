== What Is FIPS 140-3?

The Federal Information Processing Standard 140-3 (FIPS 140-3) is a U.S. government standard that defines security requirements for cryptographic modules. It specifies which algorithms are approved for protecting sensitive government data and which must be blocked.

Organizations that must meet FIPS 140-3 include those pursuing *FedRAMP* authorization, *DoD* systems under STIGs, and any federal contractor handling Controlled Unclassified Information (CUI).

Practically speaking, FIPS 140-3 compliance means:

* Only approved algorithms like SHA-256, AES-256-GCM, and HMAC-SHA256 may be used for security-critical operations
* Insecure algorithms like MD5 (used for security), SHA-1, and ChaCha20-Poly1305 must be blocked by the cryptographic provider
* The cryptographic modules themselves must be validated by NIST-accredited testing labs

== Two Levels of FIPS Compliance

Full FIPS 140-3 compliance requires enforcement at *two* distinct levels. Missing either one means your system is not fully compliant.

[NOTE]
====
*From: Nate (Senior Sysadmin)*

Hey,

Before you jump in, I want to flag something that trips up a lot of folks new to FIPS work: there are two separate things you need to get right, and they're independent of each other.

First, your container image needs to use the `-fips` variant. That swaps in the RHEL-validated FIPS provider for OpenSSL and locks down the algorithm list inside the container.

Second, your host (or cluster node) needs to boot with `fips=1` as a kernel parameter. Without that, even a FIPS image won't be "fully" compliant from a certification standpoint — though the algorithm restrictions *will* still be enforced at the container level.

In this lab environment, the host is *not* in FIPS mode (you'll confirm this in a moment). That's fine — we can still demonstrate how the container enforces algorithm restrictions independently. For a production FedRAMP deployment, you'd need both.

Good luck, and ask if anything's unclear.

-Nate

Here is what each level requires:

[cols="1,2,2"]
|===
|Level |What it covers |How to enable

|*Container level*
|Validated FIPS crypto provider inside the container; algorithm restrictions enforced for the containerized application
|Use the `-fips` image variant (e.g., `python:3.14-fips`)

|*Host/Kernel level*
|Kernel-level FIPS mode; required for full certification; affects all processes on the host
|Boot with `fips=1` kernel parameter; FIPS-validated kernel crypto modules

|===

== Step 1: Check Host FIPS State

The kernel exposes its FIPS mode status through a virtual file in `/proc`. A value of `0` means FIPS mode is not active. A value of `1` means the kernel was booted with `fips=1`.

[source,bash,role=execute]
----
cat /proc/sys/crypto/fips_enabled
----

----
0
----

As expected, the host is not in FIPS mode. This is a standard lab environment — not a hardened production system. The `0` confirms that the kernel boot parameter `fips=1` was not set.

[TIP]
====
To enable host FIPS mode on RHEL, you would run:

[source,bash]
----
fips-mode-setup --enable
reboot
----

This installs the appropriate crypto policies, sets `fips=1` in the kernel command line, and reboots. After the reboot, `cat /proc/sys/crypto/fips_enabled` would return `1`.

== Step 2: Review the Containerfiles

In the next module you will build two container images from these Containerfiles. Both images run the same Python test application — the only difference is the base image: one uses the standard `python:3.14` image, the other uses `python:3.14-fips`.

The test application (`test-fips.py`) is already in your home directory at `~/fips/test-fips.py`. It checks:

* Whether the FIPS validated provider is active in OpenSSL
* Whether FIPS-approved algorithms (SHA-256, AES-256-GCM, HMAC-SHA256) are available
* Whether disallowed algorithms (MD5, HMAC-MD5, ChaCha20-Poly1305) are blocked

=== Standard Python Containerfile

[source,bash,role=execute,subs=attributes+]
----
cat > ~/fips/Containerfile << 'EOF'
FROM {rhhi-registry}/python:3.14

COPY test-fips.py .

USER ${CONTAINER_DEFAULT_USER}
STOPSIGNAL SIGINT
ENTRYPOINT ["python", "./test-fips.py"]
EOF
----

This uses the standard Red Hat Hardened Image for Python 3.14. It includes security hardening but does not load the FIPS validated provider. Disallowed algorithms remain available to applications.

=== FIPS Python Containerfile

[source,bash,role=execute,subs=attributes+]
----
cat > ~/fips/Containerfile.fips << 'EOF'
FROM {rhhi-registry}/python:3.14-fips

COPY test-fips.py .

USER ${CONTAINER_DEFAULT_USER}
STOPSIGNAL SIGINT
ENTRYPOINT ["python", "./test-fips.py"]
EOF
----

This uses the FIPS variant of the same Python 3.14 image. The `-fips` variant:

* Includes the RHEL FIPS validated OpenSSL provider
* Configures OpenSSL to load the FIPS provider at startup
* Blocks all non-FIPS-approved algorithms, even when the host kernel is not in FIPS mode
